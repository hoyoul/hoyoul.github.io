개요

website를 만들때, 어떤식으로 설계를 해야될지 어떤 순서로 작업을 해야할 지 막막하다. 나는 web쪽에 경험이 있는것도 아니고, 그냥 기초적인 web지식만 있을 뿐이다. 어떤식으로 site를 hosting하고, 어떤 html을 만들고, 어떻게 css,js를 만들고, web framework을 쓸건지 말건지, 문서는 org모드와 연동할지 말지…dynamic or static web site를 만들건지 고려되어야 할 사항이 많다. 우선 화면부터 설계하기로 했다. 화면을 설계하기 위해서 구체적인 html과 css,js를 기술하지도 않고, 단지 어떤식으로 할지 대략적으로 design하기로 했다. 이것이 sketch라고 하고, 원래 처음에 UI/UX에 대한 abstract design을 한다.

설계

처음 부터 구체적인 설계는 아니고, symantic을 표현할 수 있게 스케치하기로 했다. 스케치는 block을 만들고, block에 이름을 달았다. 각 block이 어떻게 html이나 css로 detail하게 구현될 것인가?는 나중에 하기로 하고 모든 것을 block형태로 만들고 이름을 붙이는 방식이다. block은 어차피 div로 대체될 것이고, 임의로 지은 이름들은 class로 대체 될 것이라고 생각했기 때문이다. page는 2개의 mode가 있다. stack mode와 graph mode다. stack mode는 page의 link를 클릭하면, 새로운 화면으로 갱신되는게 아니라, 쌓여지는 방식이다. graph mode는 graph를 보여주고 click하면 해당 문서가 보여지는 형태다.

abstract design에 맞춰 css를 만든다.

html이 주어질 때, css를 어떻게 작성해야 할지 고민이 된다. css가 하는일은 element들의 appearance(size,color,font…)와 element들의 layout이라고 보여진다. element의 layout을 작성하는게 html의 요소별로 appearance보다 더 general하기 때문에 먼저 기술할 것인가? 아니면 어떤 작성하는 순서가 있는가? 괜한?고민이다.

우선 css파일을 만들기로 했다. 그냥 css파일을 만들고, abstract design시 했던 class와 id를 먼저 작성 해 보기로 했다. 그런데, 생각해보니, 그렇게 하면 안될 듯하다. css는 여러가지 특성이 있다. css요소별 우선순위에 따른 hiearchy도 있고, 여러 곳에서 style을 정의 할 수 있다. 따라서 css문서를 작성할때, 이런 것도 고려해야 할거 같다. 예를 들어서, 내가 만든 class들은 모두 div태그를 사용한다. class별로 따로 작성하는게 아니라, 공통적 내용이 있다면, div에 기술해야 한다. css는 hiearchy가 있기 때문이다. 그리고 위에서 말했듯이 css는 어느 stylesheet를 사용할지 어떤 우선순위에 맞출지도 고려해야 한다. 이것을 고려하지 않는다면 수정할때 곤란해진다. 문제가 생겼을때, 고치기 어렵게 된다. 예를 들어서, 이상한 border를 가진 박스가 보여지는데, 그것을 고치려고 한다. 이럴때, 어느 stylesheet에서 정의된것인지도 찾아야하고 tag에 정해진 style인지, 아니면 browser의 기본속성값인지, 외부 stylesheet에 정의된것인지 하나하나 다 찾아야 한다. 유지보수가 상당히 어려워 진다. 그래서 어떤 자신만의 규칙이 필요하다. 너무 복잡하게 얘기했다. css문서 template을 만들기로 했다.

css template 문서 작성

위에서 말한 abstract design에서 정의한 class와 id에 대한 속성정의 보다 먼저, css style은 여러곳에서 중복 정의될 수 있기 때문에, 나만의 규칙을 정했다. css style이 정의되는 곳은 다음과 같다. 그리고 적용되는 우선순위도 있다. 아래로 갈수록 우선순위가 높다.

  • browser stylesheet
  • 외부 stylesheet
  • style tag in html
  • inline style in tag

난 중복되서, 수정하기가 곤란할꺼 같아서 외부 stylesheet만 이용할 것이다. 그리고 style sheet내에서도 style들은 중복될 수 있다. 여기서도 적용되는 우선순위가 있다. 아래로 갈수록 우선순위가 높다.

  • *(전체선택자)
  • tag element selector
  • class selector
  • id selector

공통적인 속성은 위에 기술하고, 세부적인 속성은 아래에 적용해야 한다. 위에서 예로 들었던 class는 div태그로 만들어지기 때문에 div tag selector에서 공통적인 내용을 기술하고 class에서 세부적 내용을 기술하는 방법으로 하기로 한다. 그래서 다음과 같은 template css파일을 만들었다

** /* Project Name: /* Version: 1.0 /* Author: holy-frege /* Date: * Memo: stylesheet는 1개만 사용. style의 우선순위는 역순으로 기재 /**************************************** * ** * * [1-전체선택자] * * ** *

* ** * * [2-태그] * * ** *

* ** * * [3-class] * * ** *

* ** * * [4-id] * * ** *

그런데 이렇게 div와 class의 속성들이 나눠져서 존재하는게 css의 hiearchy한 속성은 충족 시키지만, 유지보수는 어려워보인다. stylesheet도 여러개로 나누어져 있고, hiearchy한 속성을 가지고 있었다. 그런데 나는 유지보수가 어렵다고 생각했기 때문에 하나의 문서만 사용하기로 했던 것처럼, 나는 여기서 중복이 되더라도 class 속성들은 class에서 다 처리하기로 한다. class들의 공통 tag인 div에서, 공통된 속성을 처리하지 않기로 한다. class에서 중복이 되더라도 처리한다. 그리고 class내부의 element들도 마찬가지로 중복이 되더라도,class내에서 처리하는 방식을 사용하기로 한다.

예를 들면 다음과 같다. myclass내의 h3는 아래와 같이 처리한다. h3에서 공통적인 특성을 처리할 수 있음에도 우선은 이렇게 작성하고, 나중에 refactoring 작업을 하면서 공통된 부분들을 위로 올리는 방식을 생각하자.

layout에 대해서

위에서 설명한대로 css의 기본템플릿을 만들고, 사용되는 tag와 selector를 template에 써넣은 후에 border를 보여주는 이유는 layout을 가장 먼저 처리하기로 했기 때문이다. layout은 flex를 사용할 것이다. flex는 부모가 자식들의 layout을 관리할 수 있기 때문이다. row형태로,혹은 column형태로 할수 있고, row형태라는것은 inline형태인데, inline은 content크기에 영향을 받는다면, width와 height를 설정해서 content크기와 별개로 고정되게 할수도 있기 때문이다.

header와 page container의 size와 layout 처리

  • 개요

    web browser의 화면을 보면 하얀색 도화지가 있다. 그리고 거기에 tag라는 아이템들이 놓여져 있다. 어떤 layout mananger가 있어서 자식에 해당하는 item들의 크기와 위치를 마음대로 정할 수 있다고 하자. 예를 들어서, h1 태그의 width와 height를 100x100으로 하고, 위치는 40,30에 놓겠다. div는 50x50 size에 500,30에 위치하게 하겠다. 이런 방식은 java나 python같은 gui programming에서 많이 사용하는 방식이다. 부모는 layout manager라는것을 가지고 있어서 자식들의 size와 position을 정한다. web의 html tag도 부모와 자식이 있는 hiearchy한 관계를 가지고 있다. 하지만, 위와 같은 방식과는 좀 다른 방식으로 한다. 그럴수 밖에 없는게 우선 web browser의 하얀색 창(window객체)은 도화지라기 보다 문서형태다. 다음 그림을 보자.

    그리고 각각의 tag item은 자유로운 형태를 가진게 아닌, 좀 정형화된 형태다. content에 따라 size가 정해지지만, line이 있는 note에 적용되기 때문에 각각의 tag의 size는 note에 맞게 line을 다 차지 할지, 아니면 content만큼 size를 차지할것인가를 기본 속성으로 갖기 때문이다. 여튼 모든 tag들은 정형화된 size 몇몇을 지원하는 display라는 property를 갖는다.

  • display 속성

    모든 tag는 display 속성을 가지고 있다고했다. 이 속성은 size와 layout에 영향을 미친다. 여러가지 속성값이 있지만, block,inline을 설명해 보겠다. block 속성은 width=100%가 숨겨져 있다. 그래서 content의 길이나 크기와 상관없이 line길이 만큼 갖는다. 물론 100%의 의미는 부모의 width의 100%를 의미한다. window의 line을 모두 차지 하는게 아니다. 하지만 line을 모두 차지 하지 않더라도, 다른 item이 그 line에 침범할 수는 없다. height의 경우는 content만큼만 갖는다. 반면 inline은 width와 height 모두 content에 맞게 설정된다. 그림으로 보면 아래와 같다.

    block은 width=100%가 숨겨져 있다고 했다. 그래서 div같은 tag는 default display속성값이 block이기 때문에 div는 부모의 line 전체를 다 차지한다. 근데 %라는것은 상대적인 값이다. 비교할려는 대상이 있어야 한다. 그 대상은 parent item이다. 즉 display라는 속성은 %값으로 parent의 width와 비교를 한다. 예를 들어서 html tag는 display 속성이 default로 block이다. 따라서 width=100%가 숨어있다. 그러면 html의 parent는 무엇인가? window 창이 부모가 된다. 따라서 html tag는 화면에 보이는 창의 line의 width영역을 다 차지하게 된다.

    <!doctype html>
<html lang="en" style="border:solid blue;">
this is html tag
</html>

    html tag만 있는 문서는 존재할 수 없다. browser에 의해서 해석될때 자동으로 body tag가 만들어져서 html문서가 만들어진다. 물론 html의 자식에 해당하는 “this is html tag"라는 string은 body의 item이 된다.

    여튼 중요한것은 3가지!

    1. 모든 tag는 display 속성을 가지고 있다. default로 block속성을 갖는다. html이나 body는 block속성을 갖는다.
    2. block 속성에는 width=100%가 숨겨져 있다.
    3. block속성을 갖는 모든 tag는 default로 부모의 width의 100%를 사용한다.

  • layout의 고민

    header와 page container는 container역할을 하는 div태그다. container역할을 하는 div tag는 첫번째로 자신의 size를 설정하고, 두번째로 자식 item들을 배치한다.

    header와 page container의 size를 설정하는 방법은 3가지다. pixel값으로 width와 height로 나타내거나, %로 width,height를 나타내거나, 설정하지 않고 default값을 사용하는 것이다. header와 page container의 부모는 html과 body tag다. html과 body tag의 부모는 window창이다. window창은 동적으로 변한다. device에 따라, 모니터에 따라, 사용자가 browser의 크기를 조절함에 따라 window의 크기는 달라진다. html과 body tag는 모두 display로 block속성을 default로 가지고 있다. 이말은 window창이 동적으로 변할 때,html과 body의 width는 window창의 width를 100% 사용한다. 이말은 부모 tag인 html,body는 동적인 width size와 height는 1줄에 해당하는 고정된 size를 갖는다는 얘기다. 이런 상황에서 header와 page container의 size를 설정해야 한다. header,page container tag의 size를 설정해 보자.

    1. pixel로 고정된 size로 표시하는 경우: 부모의 지배에서 벗어난다. 고정된 size로 설정하면, parent와는 아무 연관성이 없다. 부모에 해당하는 window창,html,body는 동적으로 size가 변경되는데, pixel로 고정하면 고정된 영역만 보여준다. 즉 부모의 size가 줄어들면, 보여주는 정보가 잘린다. 내가 원하는 모습은 아니다.
    2. default로 설정하는 경우: default는 width와 height property를 설정하지 않는 것이다. 그러면,default로 width:100%가 된다. 이것은 부모 width size에 영향을 받는데, 부모가 html,body,window객체라서 동적 width를 갖는다. 즉 폭이 좁아지거나 늘어나면, header와 page container도 동적으로 줄어지고 늘어난다. 부모의 width가 줄어든다고 해도, header와 page container의 모든 정보를 보여줄 수 있다는 장점이 있다. 그러나 height는 1줄로 고정된 크기를 갖는다.
    3. %로 표시하는 경우: 모든 부모의 설정사항을 살펴봐야 한다. 부모의 width,height설정에 따라서 모양이 결정된다. html,body가 pixel로 고정된 size라면 그 size안에서 비율적으로 움직인다. html,body가 default를 사용한다면, width는 부모처럼 동적으로 움직이게 된다. 반면 height는 1줄로 고정된 크기를 갖게 된다.

    어쨋든 지금 header와 page container의 size를 설정하기 위해 display속성을 default로 하기로 했다. default로 한다는 말은 width와 height를 기술하지 않는다는 것을 말한다. 이렇게 되면, div tag의 default속성인 disply: block을 사용하게 되고, block은 width:100%라는 속성값을 가지게 되고, height는 1줄을 사용하게 된다. 이것이 의미하는 것은, header나 page container는 width는 창에 따라 동적으로 움직이고, height는 content크기에 맞춰 고정되어 있다는 것이다. 단 page container의 경우 page의 크기를 1줄로 하는건 아닌거 같다. 그래서 window의 height를 그대로 전부 다 사용하기로 했다. 그렇게 하기 위해선 parent에 해당하는 html,body에서 height를 100%로 해야 한다. 그래서 child인 page에서 height를 100%로 기술해서 window의 창을 1줄이 아닌 전체를 사용할 수 있게 된다. 이것을 구현해 보자.

  • logo size변경

    먼저 logo가 계속 거슬려서 이것부터 먼저 해결한다. logo라는 img는 logo라는 div의 자식이다. image의 사이즈는 512x512이다. 부모 tag의 size를 30x30으로 하면 자식 tag인 img는 30x30으로 줄여질까? 안된다. 자식 이기는 부모 없듯이, 자식은 부모의 틀을 벗어나 그대로 512x512로 표시된다. 즉 부모와 자식간의 관계는 자식이 자신의 size를 %로 표기했을때만 고려된다.

    그래서 img tag를 30x30으로 설정했다.

  • page container와 header의 size결정

    header의 size는 별다른 기술을 하지 않는다. 반면에 page container는 height가 다르다. div에게 주어진 1줄이 아닌 window창을 전체 사용할 것이다. 그렇게 하기 위해선, height:100%로 설정하고 부모 tag의 height를 모두 height:100%로 설정해야 한다.

    html,body{
    height:100%;
}

.header{
    border: solid blue;
}
.page_container{
    border: solid #ffccaa;
    height:100%;
 }

    이렇게 하면 page container의 height는 1줄이 아닌 browser창의 height를 다 차지하게 된다.

  • layout 설정

    header와 page container는 html,body의 자식들이다. html과 body의 layout manager는 block이다. header와 page container에서 item들을 가로방향으로 나열할 것이기 때문에 flex layout을 사용하기로 했다. header와 page container에 아래와 같이 설정했다.

    display: flex;

    위 그림에서 보듯이 layout은 가로방향으로 제대로 처리 된듯 하다.

    flex에 대한 설명이 필요할 듯 하다.일반적인 얘기를 해보자. layout에 대한 얘기다. web문서에서 layout manager라는게 있을까? gui programming에서는 canvas와 같은 공간에서 직접적인 size와 position을 사용해서 layout처리를 한다던지, 부모 자식간의 관계를 통해서 부모가 layout manager가 되어서 자식 item의 배치를 다룬다. 그런데 web은 처리가 다르다. 우선 web문서에서 사용하는 공간은 note형태다. 도화지의 형태가 아니다. 따라서 절대적 좌표와 size로 appearance를 표현하지 않는다. 각각의 item은 기본적으로 note의 line에 써지는 형태다. text가 note에 써지는것과 비슷하다. 그래서 display속성에서 마치 제목처럼 block형태로 쓸것인지, 일반 sentence형태로 보여질 지를 display라는 속성을 통해서 나타낸다. 그런데 html문서는 hiearchy를 가지고 있다. parent-child관계가 tag사이에 존재한다. 이것은 programming언어에서 parent가 child item의 layout을 관장하는 방식이 web문서에도 가능할 수 있다는것을 보여준다. display가 block이나 inline같은 경우에는 부모와 독립적으로 자신의 appearance를 나타낼 뿐이였다. 다만 자신의 size를 %로 표시할 경우, 부모에 의존하게 되는데, 상대적인 비율이기 때문이다. 그런데 이것은 gui프로그래밍처럼 parent tag가 item들의 layout을 담당하는것과는 다른것이다. 독립적인 tag들이 자신의 appearance를 담당하고, 단지 부모의 size를 참조할 뿐이였다. 그런데 flex는 좀 다르다. 이것은 parent tag가 layout manager와 같은 역할을 한다. parent tag는 flex container로 부르고 자식 item들은 flex item으로 부르며, 각각의 item들이 어떤식으로 배치될지를 flex container가 결정할 수 있기 때문이다. 그래서 flex container에 display:flex라는 설정을 하면 자식 item들이 column by column 형태로 배치된다. flex container에서 flex-direction:column으로 하면 item들은 row by row로 배치된다. 또한 flex container공간내에서 item들을 왼쪽부터 나열할지 오른쪽부터 나열할지를 justify-content로 정할 수도 있고, align-items로 item들을 위 아래 방향에서 어떻게 나타낼지를 정할 수도 있다. layout manager의 기능을 하는것이다. 이것에 대해선 자세한 설명이 아닌, 이정도 개념만 알기로 한다.

  • page의 size조정

    page container에 들어갈 page는 width:100%와 height:100%로 설정되어 있으면 안된다. 그러면 page가 browser의 화면창에 꽉차게 된다. 내가 원하는건, 화면창이 640px 이상이라면 첫번째 page는 640px의 크기를 갖고, 만일 브라우저 창이 640px이하라면 창에 맞춰서 보여주는 것이다. 이것을 하기 위해서 다음과 같이 설정했다.

    .page{
    max-width: 640px;
    width:100%;
    border: solid #ffdd00;
}

    물론, 이것은 media query를 사용하면 된다. 하지만, 우선 그냥 이런식으로도 할 수있다. 뭐 그런거임.

org파일을 md파일로 변환하는것은 emacs의 org-hugo-export-wim-to-md라는 function을 사용하면 된다. 이 function은 ox-hugo 라이브러리에 있는데, 수백, 수천개의 org파일을 md파일로 바꿀때 이함수를 쓴다. 그런데, 수백, 수천의 파일을 md파일로 변경하는 작업은 elisp을 사용하여 프로그램으로 돌리면 시간이 너무 많이 걸린다. 그래서 make같은 build tool을 써야 한다. 수백개의 파일을 처리하는것은 programming으로 하는것 보다, build tool을 쓰는게 훨씬 좋다. 왜냐하면 incremental builds라는 기능 때문이다. 이 기능의 장점은 이미 변경된것은 다시 변경하지 않는다는 것이다. 예를 들면, 100개파일중 50개가 md로 변경된 이후, build를 하면 변경되지 않은 것 50개만 build한다. 이 기능때문에 program으로 대량의 파일에대한 처리를 하지 않고, build tool을 사용해야만 했다. make대신 ninja를 선택한것은 내가 참고로한 braindump의 소스도 make를 사용하기 위해선 cmake도 공부해야한다는게 걸렸다. 그리고 이 사이트에서 너무나 간단하다고 해서 선택하기로 했다. 그런데, 이 사이트에서 ninja의 단점으로 wildcard같은 pattern을 사용하지 못한다고 해서 좀 후회했다. 왜냐면, 특정 폴더에 있는 대량의 파일을 변환하기 위해서 ninja를 사용하는건데, ninja의 script에서는 wildcard를 사용하지 못한다는 것은 정의한 rule을 대량의 파일에 적용하지 못한다는 얘기기 때문이다. 즉 for-loop을 돌릴수 없는것이다. ninja는 그래서 cmake처럼 일종의 generator를 만들어서 대량의 파일을 처리할 수 있게 한다. 그것이 위의 참조 사이트에도 나와 있지만, python같은 program내에서 ninja의 script를 만들고 ninja를 실행하는 방법을 사용한다. 나도 동일한 방식을 사용하기로 했다.

ninja로 org->md

build.py의 작성

내가 org-roam으로 작성할 문서는 org-roam-directory에 지정된 곳에 저장된다. org폴더라고 하자. 여기에 있는 org file들을 md로 변환해서 hugo-base-dir의 content/posts폴더안에 넣을려고 한다. 그런데 ninja는 pattern이 제공되지 않는다. 파일의 수가 작다면, build.ninja에서 rule과 build만 적의하면 되지만, 대량의 파일을 처리하기 위해선 program의 도움을 받아야 한다. python으로 이것을 구현한다. 아래는 ninja를 사용하는 python의 전형적인 형태다. 즉 build.ninja를 정의한 후에 ninja를 실행하는 구조다. rule은 shell에서 사용하는 명령어가 아니고 emacs의 ox-hugo에 있는 함수를 사용한다. 그렇기 때문에 rule은 emacs를 실행해서 특정함수에 org파일을 전달해서 md파일로 변환하는 형식을 사용한다. 이 부분은 org2md.el이라는 file에 별도로 기술했다.

#!/usr/bin/env python
import glob
from pathlib import Path

files = glob.glob("org/*.org")

# rule을 정의 [build.ninja의 작성]
with open('build.ninja', 'w') as ninja_file:
    ninja_file.write("""rule org2md
  command = emacs -Q --batch -l org2md.el --eval \"(holy/publish_md \\"$in\\")"
  description = org2md $in
""")
# build를 org폴더의 org파일 개수만큼 생성 [build.ninja 작성]
    for f in files:
	path = Path(f)
	output_file = f"content/posts/{path.with_suffix('.md').name}"
	ninja_file.write(f"""
build {output_file}: org2md {path}
""")

# 지금껏 작성한 build.ninja를 실행
import subprocess
subprocess.call(["ninja"])

org2md.el의 작성

ox-hugo의 function을 사용해서 org문서를 md로 변환한다. emacs에서 ox-hugo의 function을 사용하기 위해서는 org가 dependent하기 때문에 org와 ox-hugo를 설치해야 한다. straight.el을 사용해서 install과 load를 하기로 했다. 왜냐하면, ox-hugo는 melpa같은 저장소에 있지 않기 때문이다. github에서 유지관리 되기 때문에, github에서 가져와서 설치및 load를 위해 straight.el을 사용하기로 했다. 모든 org파일을 md로 변경할때 emacs instance를 실행하고 초기화 파일은 기존 local system의 init.el을 load하면 시간이 많이 걸리기 때문에 org2md.el이란 파일만 load해서 사용한다. 그렇게 하기위해서, -Q옵션을 사용하고, org2md.el에서는 위에서 말한 straight.el을 설치한 후 org와 ox-hugo를 load하고 holy/publish라는 함수를 정의해서 인자로 org파일이 들어오면 md파일로 변환해서 저장하는 일을 하게 된다. 소스를 보면 다음과 같다.

;; use-package대신 straight.el을 설치,왜냐면 ox-hugo는 melpa같은 repo에 없고 github에서 관리되는 package이기 때문
(defvar bootstrap-version)
(defvar straight-base-dir)
(defvar straight-fix-org)
(defvar straight-vc-git-default-clone-depth 1)
(defvar publish--straight-repos-dir)

(setq gc-cons-threshold 83886080 ; 80MiB
      straight-base-dir (expand-file-name "../.." (or load-file-name buffer-file-name))
      straight-fix-org t
      straight-vc-git-default-clone-depth 1
      publish--straight-repos-dir (expand-file-name "straight/repos/" straight-base-dir))

(let ((bootstrap-file
       (expand-file-name "straight/repos/straight.el/bootstrap.el" user-emacs-directory))
      (bootstrap-version 5))
  (unless (file-exists-p bootstrap-file)
    (with-current-buffer
	(url-retrieve-synchronously
	 "https://raw.githubusercontent.com/raxod502/straight.el/develop/install.el"
	 'silent 'inhibit-cookies)
      (goto-char (point-max))
      (eval-print-last-sexp)))
  (load bootstrap-file nil 'nomessage))

;; straight.el로 org와 ox-hugo설치
;; (straight-use-package
;;  '(org-mode :type git
;;             :host github
;;             :repo "emacs-straight/org-mode"
;;             :files ("*.el" "lisp/*.el" "contrib/lisp/*.el")))
(straight-use-package 'org)
(straight-use-package
 '(ox-hugo :type git
	   :host github
	   :repo "kaushalmodi/ox-hugo"
	   :nonrecursive t))

(require 'find-lisp)

(setq org-roam-directory "/Users/holy/MyProjects/holy2frege.github.io/org")
(setq org-hugo-base-dir "..")

(defun holy/publish_md (file)
    (with-current-buffer (find-file-noselect file)
      (message (format "[build] Exporting %s" file))
      (org-hugo-export-wim-to-md :all-subtrees nil nil nil)))

(message "Done!")

holy/publish가 ninja의 rule에서 사용하는 함수다. 함수가 정의되기 이전의 코드는 straight.el을 설치하고, org와 ox-hugo를 설치하고 load하는 작업이다. holy/publish_md의 동작은 간단하다. with-current-buffer는 인자로 buffer를 받아서 org-hugo-export-wim-to-md로 md 파일을 만든다. org buffer에서 M-x org-hugo-export-wim-to-md를 실행하면 md파일이 만들어지는것과 같다. find-file-noselect는 유명한 find-file함수와 동일하다. 즉 file을 open해서 buffer를 만든다. C-x f라는 명령으로 file을 찾아서 buffer에 보이게 하는것과 동일하다. 차이점이 있다면, buffer는 일반적으로 화면에 보인다. 그런데 find-file-noselect는 file을 열어서 buffer객체만 만들 뿐 화면에는 보이지 않는다.

폰트에 대해서 [css]

layout을 설정한 후에 난 font를 설정한다. page에 쓰여지는 모든 글자의 font(글꼴)는 Inter font를 사용한다. 그래서 inter font를 google에서 가져와서 설치할 수 있게 하는 코드를 다음과 같이 설정한다.

/* =============================================================================
   everything
   ========================================================================== */

@import url('https://fonts.googleapis.com/css2?family=Inter');

이렇게 하면 이 font가 없는 client는 이 폰트를 다운 받는다. 그리고 body에서 이 폰트를 설정한다. body에 설정하는 이유는 body에 쓰여지는 모든 글자를 inter로 하기 위함이다.
